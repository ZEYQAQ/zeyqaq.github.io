三下两下就改完了，跑了一遍，没啥问题，第二天早上就正确告警出来了。

脚本功能：堆内，JVM参数配置OOM自动打dump，dump转储到并行文件系统（参考其他云的存储桶功能），使用云厂商工具产生dump文件下载链接，调用IM的webhook自动告警至运维研发群，附带下载链接；堆外，通过oom_reason=$(kubectl get pod "$pd" -o jsonpath='{.status.containerStatuses[0].lastState.terminated.reason}')，筛选"OOMKilled"并读取终止时间去重，与历史记录比对，不同则告警KILL信息。



很正常的一次排查，但是有个匪夷所思的事情：脚本一次运行就告警：某工作负载堆外原因，第2227次重启。

注，-Xmx：最大堆内存，-Xms：最小堆内存，-Xss：每个线程的栈大小

| 参数                        | 作用             | 不设的后果                                     |
| :-------------------------- | :--------------- | :--------------------------------------------- |
| `-XX:MaxMetaspaceSize`      | 类元数据区上限   | **无上限**，加载的类越多占用越大，可以吃光内存 |
| `-XX:MaxDirectMemorySize`   | NIO 直接内存上限 | **默认等于 -Xmx**，理论上可以再吃 6GB          |
| `-XX:ReservedCodeCacheSize` | JIT 编译缓存     | 默认 240MB，一般够                             |

配置口诀：

```
Xmx ≤ 75% × limits          堆不超过容器限制的 75%
limits = Xmx + 2Gi          给堆外留 2Gi 左右的余量
requests = Xmx               调度预留至少等于堆大小
Xms = Xmx                   容器里不要动态伸缩堆
HeapDumpPath = PV 路径       dump 写到持久卷，重启不丢
MaxMetaspaceSize 必须设       堵住无限增长的口子
MaxDirectMemorySize 必须设    堵住无限增长的口子
```
